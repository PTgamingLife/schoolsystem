# 教室預訂系統規劃與下一步

## 第一版已完成

- 使用者預訂流程：搜尋教室、選擇日期時段、填寫個人資料、送出申請。
- 管理後台：檢視訂單、依狀態篩選、核准訂單、匯出表單入口。
- 視覺化管理：教室週使用率長條圖、時段矩陣、狀態標籤。
- 響應式介面：桌面與手機版皆可使用。

## 正式系統建議流程

1. 需求確認
   - 確認可預訂教室清單、校區、建物、設備、容納人數。
   - 確認簽核層級，例如承辦人、單位主管、總務處。
   - 確認表單格式，例如 PDF、Google 表單、校內公文系統或 Email 附件。

2. 資料庫設計
   - users：申請人與管理者帳號。
   - classrooms：教室主檔、設備、容量、是否開放。
   - bookings：預訂訂單、日期、時段、用途、人數、狀態。
   - approval_logs：簽核紀錄、處理人、時間、意見。
   - attachments：補件資料或匯出表單紀錄。

3. 身分驗證
   - 建議優先確認學校是否提供 SSO、LDAP、OAuth 或校務系統登入。
   - 若第一版先內部使用，可先做管理者帳密登入與權限分級。

4. 後端 API
   - GET /classrooms：查詢教室。
   - GET /availability：查詢可預訂時段。
   - POST /bookings：建立預訂申請。
   - PATCH /bookings/:id：管理者修改資料或狀態。
   - POST /bookings/:id/export：產生簽核表單。

5. 表單與簽核
   - 系統送出後自動產生表單編號。
   - 狀態至少包含：待簽核、已核准、退回補件、已取消。
   - 每次狀態變更都要留下處理人、時間與備註。

6. 通知
   - 申請送出：通知申請人與承辦人。
   - 核准或退回：通知申請人。
   - 可用 Email、校內通知、LINE Notify 替代方案或其他既有管道。

7. 上線與維運
   - 先用測試資料讓行政端試跑一週。
   - 確認衝突預訂、重複送出、補件流程、匯出表單格式。
   - 再接正式資料庫與校內登入。

## 建議 MVP 範圍

先做「單一校區、固定時段、行政後台人工簽核」版本。這樣可以快速替代人工收件與 Excel 管理，等流程穩定後再加入跨校區、借用規則、黑名單、定期預訂與自動簽核。
